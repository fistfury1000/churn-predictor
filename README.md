# Customer Churn Prediction with Explainable AI

A binary classification model that predicts whether a telecom customer will
churn, served via a FastAPI endpoint that returns not just a risk score but
the **top factors driving each individual prediction** using SHAP.

## Problem Statement

Customer churn is one of the most direct revenue levers in subscription
businesses. A model that only outputs "will churn / won't churn" isn't very
actionable for a retention team — they need to know *why*, on a per-customer
basis, so they can intervene with the right offer. This project treats
explainability as a first-class requirement, not an afterthought.

## Dataset

[Telco Customer Churn](https://www.kaggle.com/datasets/blastchar/telco-customer-churn)
(IBM sample dataset, via Kaggle) — 7,043 customers, 19 features covering
demographics, account info, and services subscribed to. Target: `Churn`
(Yes/No), ~27% positive class.

## Approach

1. **Preprocessing** (`src/data_loader.py`)
   - Coerce `TotalCharges` to numeric (blank strings for tenure-0 customers)
   - Label-encode categorical features
   - No scaling needed — tree-based models are invariant to feature scale

2. **Modeling** (`src/train.py`)
   - XGBoost classifier, tuned for mild class imbalance via `scale_pos_weight`
   - 80/20 stratified train/test split
   - Evaluated on accuracy, precision, recall, F1, ROC AUC — recall matters
     more than accuracy here, since missing a churner is costlier than a
     false alarm

3. **Explainability** (`src/explain.py`)
   - `shap.TreeExplainer` computes per-prediction feature attributions
   - API surfaces the top 3 factors (by absolute SHAP value) for every
     prediction, in plain language

4. **Deployment** (`api/main.py`)
   - FastAPI service, single `/predict` endpoint
   - Takes raw customer features (not pre-encoded), handles encoding
     internally, returns probability + risk tier + top factors

## Results

Trained on the full Telco Customer Churn dataset (7,043 customers, 80/20 stratified split).

| Metric    | Score  |
|-----------|--------|
| Accuracy  | 0.7566 |
| Precision | 0.5283 |
| Recall    | 0.7727 |
| F1        | 0.6276 |
| ROC AUC   | 0.8410 |

**Key drivers of churn** (top features by XGBoost gain-based importance):
1. **Contract type** — by far the strongest predictor (0.37 importance, roughly 3x the next feature)
2. **Online Security** — customers without it churn more
3. **Tech Support** — same pattern as Online Security
4. Internet Service type, Streaming Movies, and tenure round out the top 6

The model was deliberately tuned to favor recall over precision (via
`scale_pos_weight`), since in a churn context the cost of missing an
at-risk customer (false negative) is typically higher than the cost of
flagging a customer who was going to stay anyway (false positive).

## Setup

```bash
python -m venv venv
venv\Scripts\activate          # Windows
pip install -r requirements.txt
```

Download `WA_Fn-UseC_-Telco-Customer-Churn.csv` from
[Kaggle](https://www.kaggle.com/datasets/blastchar/telco-customer-churn),
rename it to `telco_churn.csv`, and place it in `data/`.

## Usage

**1. Train the model:**
```bash
python -m src.train
```
This prints evaluation metrics and saves the model, encoders, feature list,
and SHAP explainer to `models/`.

**2. Run the API:**
```bash
uvicorn api.main:app --reload
```

**3. Make a prediction:**
```bash
curl -X POST http://127.0.0.1:8000/predict \
  -H "Content-Type: application/json" \
  -d '{
    "gender": "Female",
    "SeniorCitizen": 0,
    "Partner": "Yes",
    "Dependents": "No",
    "tenure": 2,
    "PhoneService": "Yes",
    "MultipleLines": "No",
    "InternetService": "Fiber optic",
    "OnlineSecurity": "No",
    "OnlineBackup": "No",
    "DeviceProtection": "No",
    "TechSupport": "No",
    "StreamingTV": "Yes",
    "StreamingMovies": "No",
    "Contract": "Month-to-month",
    "PaperlessBilling": "Yes",
    "PaymentMethod": "Electronic check",
    "MonthlyCharges": 85.5,
    "TotalCharges": 171.0
  }'
```

**Example response:**
```json
{
  "churn_probability": 0.812,
  "risk_level": "High",
  "top_factors": [
    {"feature": "Contract", "value": "Month-to-month", "shap_value": 0.31, "impact": "+0.310 (increases risk)"},
    {"feature": "tenure", "value": 2, "shap_value": 0.18, "impact": "+0.180 (increases risk)"},
    {"feature": "TechSupport", "value": "No", "shap_value": 0.12, "impact": "+0.120 (increases risk)"}
  ]
}
```

Interactive API docs (Swagger UI) are auto-generated at
`http://127.0.0.1:8000/docs`.

**4. (Optional) Run the Streamlit front end:**

With the API still running in one terminal, open a second terminal:
```bash
streamlit run app.py
```
This opens a browser UI at `http://localhost:8501` with a form for entering
a customer's profile and a live-updating churn risk + SHAP factor
breakdown. It's a thin client — all it does is call the `/predict`
endpoint, so the API must be running first.
